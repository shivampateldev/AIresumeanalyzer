SECTION 1 — PROJECT ARCHITECTURE OVERVIEW
===============================================================================

WHY A MODULE-BASED ARCHITECTURE?
  The project splits every concern into a separate Python file inside modules/.
  This is called "separation of concerns" — each module does one job only.
  Why? Because:
    - If the PDF parser breaks, you fix only resume_parser.py
    - If you want to swap Groq for OpenAI, you change only groq_ai_client.py
    - Unit testing becomes possible (test one module at a time)
    - Multiple developers can work simultaneously without merge conflicts

THE PIPELINE PATTERN
  The system uses a Linear Pipeline:
    Input → Stage1 → Stage2 → ... → Output
  Each stage receives data from the previous stage and passes enriched data
  to the next. This is the same pattern used in data engineering (ETL pipelines)
  and Unix shell piping (command1 | command2 | command3).

  In Python code this looks like:
    resume_text   = parser.extract_text(resume_input)
    raw_skills    = extractor.extract_skills(resume_text)
    norm_skills   = normalizer.normalize_list(raw_skills)
    role_skills   = scraper.fetch_skills(role, company)
    gap           = analyzer.analyze(norm_skills, role_skills)
    score         = model.predict(...)
    report        = generator.print_report(...)

--- 2.1 CLASSES AND INSTANCES ---

  Every module uses a Python class. Example:
    class SkillExtractor:
        def __init__(self, groq_client=None):
            ...
        def extract_skills(self, text):
            ...

  WHY CLASSES?
    - State persistence: extractor keeps its vocabulary loaded in memory
    - Reuse: create one extractor, call extract_skills() many times
    - Configuration: pass groq_client at construction, not every call

  INSTANCE vs CLASS method:
    Instance method:   def extract_skills(self, text)   → uses self.vocabulary
    Static method:     @staticmethod def generate(...)  → no self, pure function
    Class method:      @classmethod def create(cls)     → receives the class itself

  ReportGenerator uses @staticmethod because it has no internal state to keep.

  WHY TYPE HINTS?

--- 2.3 F-STRINGS ---

  All string formatting uses f-strings (Python 3.6+):
    name = "Python"
    print(f"I love {name}")         → "I love Python"
    print(f"{score:.3f}")           → "0.210"  (3 decimal places)
    print(f"{text:<25}")            → left-align in 25 chars
    print(f"{number:>3}")           → right-align in 3 chars

  WHY F-STRINGS over .format() or %?
    - Faster at runtime (compiled to bytecode)
    - More readable (expression inline with text)
    - Support complex expressions: f"{len(skills)} skills found"

--- 2.4 LIST COMPREHENSIONS ---

  Instead of:
    result = []
    for s in skills:
        if len(s) > 2:
            result.append(s.lower())

  We write:
    result = [s.lower() for s in skills if len(s) > 2]

  This is a list comprehension. It produces a new list by transforming each
  element. It is faster than a for-loop because it avoids Python's function
  call overhead for list.append() on every iteration.

  SET COMPREHENSION:
    skill_set = {s.lower() for s in skills}
    → produces a set (no duplicates, O(1) lookup)

  DICT COMPREHENSION:
    counts = {skill: freq for skill, freq in counter.most_common(20)}

--- 2.5 SET OPERATIONS ---

  Sets are unordered collections with NO duplicates and O(1) membership test.
  The skill gap analysis relies heavily on set math:

    resume = {"python", "git", "mysql"}
    role   = {"python", "docker", "sql", "git"}

    matching = resume & role           → {"python", "git"}  (intersection)
    missing  = role - resume           → {"docker", "sql"}   (difference)
    extra    = resume - role           → {"mysql"}           (difference)
    all_skills = resume | role         → {"python", "git", "mysql", "docker", "sql"}

  WHY SETS NOT LISTS?
    - list.index("python") → O(n) scan
    - "python" in my_set   → O(1) hash lookup
    For 300 skills this is 300x faster.

--- 2.6 EXCEPTION HANDLING ---

  Pattern used throughout:
    try:
        from pdfminer.high_level import extract_text
        text = extract_text(path)
    except ImportError:
        logger.warning("pdfminer not installed")
        return ""
    except Exception as e:
        logger.error(f"pdfminer failed: {e}")
        return ""

  ImportError  — library not installed
  FileNotFoundError — file path wrong
  Exception    — catch-all (use sparingly, always log the error)
  The pattern: try the operation, handle specific errors, fall back gracefully.

--- 2.7 *args AND **kwargs ---

    def print_report(self, target_role, company, resume_skills, ...):

--- 3.1 re (Regular Expressions — BUILT-IN) ---

  Regex is a pattern matching language for strings. Python's built-in re
  module implements it.

  PATTERNS USED IN THIS PROJECT:
    r"\bpython\b"         → word boundary match (won't match "cpython")
    r"\bc\+\+\b"          → matches "C++" literally (\ escapes the +)
    r"(?<![a-z0-9\+])c\+\+(?![a-z0-9])"
                          → negative lookbehind: c++ NOT preceded by a-z,0-9,+
    r"(\d+)\s*years?"     → capture group: 1+ digits then optional "years"

  KEY re FUNCTIONS:
    re.search(pattern, text)   → finds FIRST match anywhere in string
    re.findall(pattern, text)  → returns ALL matches as list
    re.sub(pattern, repl, text)→ replace all matches with repl
    re.compile(pattern)        → pre-compile for repeated use (faster)

  WHY COMPILE?
    pattern = re.compile(r"\bpython\b")
    # used 10,000 times in a loop:
    for text in corpus:
        pattern.search(text)
    vs:
    for text in corpus:
        re.search(r"\bpython\b", text)   # recompiles each time

  re.compile() saves 30-50% time in loops.

  LOOKAHEAD/LOOKBEHIND:
    (?<!X)Y   → Y not preceded by X  (negative lookbehind)
    (?!X)Y    → Y not followed by X  (negative lookahead)
    (?=X)     → position followed by X (positive lookahead)
    Used to match C++ without matching the C inside "architect"

--- 3.4 math (BUILT-IN) ---

  math.log(x)      → natural logarithm (base e)
  math.log(x, b)   → logarithm base b
  math.exp(x)       → e^x (inverse of log)

  PERSONALITY ANALYZER uses log scaling:
    score = math.log(hits + 1) / math.log(threshold + 1) * 10

  WHY LOG SCALE?
    Linear:  2 keywords → 2/30 * 10 = 0.67/10 (too low)
    Log:     2 keywords → log(3)/log(7) * 10 ≈ 5.7/10 (realistic)
    Log compresses large ranges and gives credit for even 1-2 hits.

  PROBABILITY ESTIMATOR uses logistic (sigmoid) function:
    p = 1 / (1 + math.exp(-logit))
    Where logit = a*score + b*match - c*competition + bias
    This maps any real number to [0, 1] — a probability.
    Why sigmoid? It's differentiable, bounded, and models probability naturally.

--- 4.1 HYBRID SKILL EXTRACTION (4-PASS PIPELINE) ---

  WHY 4 passes? No single method catches everything:
    Pass 0: ORIGINAL text   → C++, C# BEFORE lowercasing breaks them
    Pass 1: Regex patterns  → 40+ explicit patterns (node.js, .net, mysql...)
    Pass 2: Vocabulary      → skill_categories.json, 300+ skills
    Pass 3: Groq AI         → catches what regex missed (collocations, context)

  THE C++ PROBLEM:
    If you lowercase "C++" you get "c++".
    If you then run regex r"\b[a-z]+\b" you get "c" (stopped at "+").
    Solution: match C++ on the ORIGINAL text before any cleaning.

  LOOKBEHIND/LOOKAHEAD for C++:
    Pattern: r"(?<![a-z0-9\+])c\+\+(?![a-z0-9])"
    This says: match "c++" only if not preceded by alphanumeric or plus,
    and not followed by alphanumeric.
    Without this, "vc++" would match the c++ inside it.

--- 4.2 ALIAS NORMALIZATION ---

  The problem:
    Resume says "k8s". Role requires "kubernetes".
    Without normalization, they don't match even though they're the same thing.

  Solution — skill_aliases.json:
    {
      "k8s": "kubernetes",
      "sklearn": "scikit-learn",
      "postgres": "postgresql",
      "tf": "tensorflow"
    }

  normalize_list() applies this mapping to every skill before comparison.
  This is called canonical form normalization.

--- 4.3 SKILL GAP ANALYSIS (SET MATHEMATICS) ---

  This is pure discrete mathematics:
    Let R = resume_skills  (set)
    Let J = role_skills    (set)

    Matching = R ∩ J       (intersection — in both)
    Missing  = J - R       (difference — in role but not resume)
    Extra    = R - J       (difference — in resume but not role)
    Coverage = |R ∩ J| / |J| × 100%

  Example:
    R = {python, git, mysql, html}
    J = {python, docker, sql, git, machine learning}
    Matching = {python, git}     → 2 skills
    Missing  = {docker, sql, machine learning}  → candidate must learn
    Extra    = {mysql, html}     → bonus, not penalized
    Coverage = 2/5 = 40%

--- 4.4 REGRESSION MODEL FOR SCORING ---

  Features (inputs) used:
    1. skill_match_ratio    = matching / role_skills  (0-1)
    2. missing_skill_ratio  = missing / role_skills   (0-1)
    3. skill_count          = total resume skills     (int)
    4. keyword_density      = skill tokens / total words (0-1)
    5. experience_signal    = years+seniority+verbs   (0-1)

  The model was "trained" on 12 calibration points:
    (0.0 match, 1.0 missing, 2 skills) → score 0.15  (beginner)
    (0.5 match, 0.5 missing, 12 skills)→ score 0.55  (mid)
    (0.9 match, 0.1 missing, 25 skills)→ score 0.90  (expert)
  etc.

  WHY NOT DEEP LEARNING?
    We have ~12 data points. Deep learning needs thousands.
    Linear regression with 5 features is appropriate and interpretable.
    The score is also clamped: max(0, min(1, score)).

--- 4.5 THE LOGISTIC FUNCTION FOR PROBABILITY ---

  Formula:
    probability = 1 / (1 + e^(-(a*S + b*M + c*E - d*C + bias)))

  Where:
    S = resume_score  (from regression model)
    M = skill_match_ratio
    E = experience_signal
    C = competition_factor (how hard the role is to get)
    a,b,c,d = manually calibrated weights
    bias = offset to center the probability realistically

  WHY LOGISTIC?
    - Output is always between 0 and 1 (valid probability)
    - Non-linear: going from 0.8 to 0.9 score matters more than 0.1 to 0.2
    - Standard tool for binary classification problems

--- 4.6 LOG SCALING IN PERSONALITY ANALYZER ---

  The keyword lists have 25-30 words each.
  A typical 2-year-experience resume might match 4-6 of them.

  Problem with linear:    4/30 * 10 = 1.3/10 (too low)
  Problem with log:       log(5)/log(7) * 10 = 8.2/10 (too high if threshold=7)

  Our formula:
    threshold = 0.20 * len(keywords)    → 20% hit rate = "full" score
    score = log(hits+1) / log(threshold+1) * 10 * weight

  Result: 2 hits → ~5/10, 5 hits → ~8/10, 10+ hits → 10/10

--- 4.7 SKILL PRIORITY RANKING ---

  Formula:
    priority = 0.50 * job_importance
             + 0.30 * inverse_learning_time
             + 0.20 * market_demand

  job_importance   = position in role_skills list (first = most important)
  learning_time    = estimated weeks to learn (shorter = higher priority)
  market_demand    = probability of appearing on job boards

  WHY WEIGHTS 0.5, 0.3, 0.2?
    Job importance matters most (employer-defined requirement).
    Learning speed helps candidates prioritize quick wins.
    Market demand ensures they learn industry-standard tools.

--- 5.1 FACTORY / DEPENDENCY INJECTION ---

  main.py creates all objects and injects dependencies:
    groq   = GroqAIClient()
    extractor = SkillExtractor(groq_client=groq)    ← Groq injected
    scraper   = InternetJobScraper(skill_extractor=extractor, groq_client=groq)

  WHY INJECTION?
    SkillExtractor doesn't create Groq. It RECEIVES Groq.
    This means you can pass a MockGroqClient in tests.
    It also means SkillExtractor works WITHOUT Groq (just skip Pass 3).

--- 5.2 STRATEGY PATTERN (FALLBACK CHAIN) ---

  InternetJobScraper.fetch_skills():
    1. Try cache
    2. Try DuckDuckGo scraping
    3. Try Groq AI
    4. Return []

  Each "strategy" is tried in order. The system degrades gracefully.
  If strategy 1 works, 2 and 3 are never called.
  If all fail, the pipeline validation catches the empty list.

--- 5.3 SINGLETON-LIKE INIT ---

  All heavy objects are created ONCE in main() before the analysis loop:
    skill_extractor = SkillExtractor()  # loads 300 skills once
    scoring_model   = RegressionScoringModel()  # fits model once

  Then reused across multiple resumes if user runs multiple analyses.
  WHY? Loading vocabulary and fitting models takes 200-500ms.
  Doing it every analysis would add 1+ second per run.

--- 5.4 CACHING PATTERN ---

  CacheManager stores scraped results keyed by (role, company):
    cache_key = f"{role.lower()}_{company.lower()}"
    If key exists AND < 24 hours old → return cached result
    Else → scrape, store result

  WHY 24 HOURS?
    Job requirements don't change hourly.
    Scraping the same query repeatedly wastes bandwidth and risks IP blocks.
    24h cache is fast but stays fresh.

    WHY? Without timeout, a hung server blocks the program.
    12 seconds is long enough for slow servers, short enough to stay responsive.

SECTION 7 — ENCODING AND WINDOWS COMPATIBILITY
===============================================================================

  WINDOWS TERMINAL ENCODING PROBLEM:
    Windows PowerShell default encoding = CP1252 (Western European subset)
    CP1252 cannot represent: ★ ✓ ✗ → ← 📅 🎓 (Unicode outside Latin-1)
    Printing these crashes with UnicodeEncodeError.

  OUR SOLUTION:
    All terminal output uses ASCII only:
      ★ → *
      ✓ → [HAVE]
      ✗ → [MISS]
      → → ->
      📅 → >>
    The .txt file is saved with encoding="utf-8" which supports all chars.

  READING FILES SAFELY:
    open(path, encoding="utf-8", errors="replace")
    errors="replace" → replace undecodable bytes with "?" instead of crashing.

  SAFE PRINT PATTERN:
    try:
        print(text)
    except UnicodeEncodeError:
        safe = text.encode("ascii", errors="replace").decode("ascii")
        print(safe)

SECTION 10 — CHARTS: HOW THEY WORK
===============================================================================

  RADAR CHART:
    1. Get skill categories for role and resume
    2. Count skills per category
    3. Create N=len(categories) equally-spaced angles around 360°
       angles = [2*pi*i/N for i in range(N)]
    4. Close the polygon by repeating the first point at the end
    5. Plot role as one filled polygon (cyan, 25% opacity)
    6. Plot resume as another filled polygon (green, 25% opacity)
    7. Label each axis with category name
    8. Dark background (#1a1a2e) → modern dark theme

  SPIDER CHART:
    Same technique, but:
    - 8 axes = 8 personality competencies
    - One polygon = candidate's competency profile
    - Axes labeled: Technical Depth, Problem Solving, etc.

  SAVING:
    plt.savefig(path, dpi=150, bbox_inches="tight", facecolor=fig.get_facecolor())
    dpi=150: resolution (150 dots per inch = sharp on screen and print)
    bbox_inches="tight": remove whitespace padding around chart
    plt.close(fig): CRITICAL — releases matplotlib memory. Without this,
      hundreds of hidden figures accumulate causing memory leak.


===============================================================================
SECTION 11 — ERROR HANDLING PHILOSOPHY
===============================================================================

  LEVELS OF ERROR HANDLING:

  1. VALIDATION CHECKPOINT (before stage):
       if len(resume_skills) < 2:
           abort with helpful message
     Prevents invalid data from propagating and producing wrong results.

  2. GRACEFUL DEGRADATION (inside stage):
       try:
           text = pdfminer.extract(path)
       except Exception:
           text = pypdf2.extract(path)   # fallback
     Stage completes, just with less-good method.

  3. SAFE DEFAULTS (post-failure):
       score = model.predict(...)
       if score is None:
           show "Score not computed" in report
     Never show random/wrong numbers. Show "unavailable" instead.

  4. LOGGING (always):
       logger.warning(f"pdfminer failed: {e}")
     Every exception is logged. Logs written to app.log.
     User sees clean messages. Developer sees full traceback in log.
